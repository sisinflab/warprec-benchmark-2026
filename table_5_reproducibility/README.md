# Table 5 Reproducibility - WarpRec RecSys 2026

This folder contains the reproducibility package for Table 5 of the WarpRec paper. It trains **ItemKNN** and **LightGCN** on **MovieLens-1M** in each of three recommendation frameworks and cross-evaluates every recommendation file in every framework (9 train × eval cells per model, 18 in total), regenerating the paper's Table 5 as LaTeX and Markdown.

## Overview

The package drives the following frameworks from a single `Makefile`:

- **Elliot**
- **RecBole**
- **WarpRec**

Each framework is installed into its own isolated micromamba environment. The Makefile calls each framework via `micromamba run -n <env>`, so no manual activation is ever required.

## Repository Structure

```
table_5_reproducibility/
├── Makefile                  # central driver; run `make help`
├── environments/             # 6 YAMLs (framework × cpu|gpu)
├── frameworks/               # per-framework assets
│   ├── elliot/               # 1 training + 4 cross-eval YAMLs
│   ├── recbole/              # training + cross-eval Python entry points
│   └── warprec/              # 1 training + 4 cross-eval YAMLs
├── scripts/                  # orchestration helpers (stdlib-only)
└── table.tex                 # regenerated by `make aggregate`
```

The pipeline additionally produces `data/`, `recs/`, `metrics/`, `experiment/`, and `results/` (all gitignored), and a `table.md` alongside the regenerated `table.tex`.

## Prerequisites

Before running the pipeline, ensure that:

1. **Micromamba** is installed and on `PATH`
2. **`make`** and **`git`** are available
3. At least **3 GB of free disk** is available for environments, the dataset, and intermediate outputs
4. (GPU only) an **NVIDIA GPU with CUDA 12.6 drivers** is attached - this matches the stack used to produce the paper's numbers

## Environment Setup

Each framework requires its own isolated environment, selected via a YAML under [`environments/`](./environments):

| Framework | Python | Deep learning    | Ray      | Install source                                 |
|-----------|--------|------------------|----------|-------------------------------------------------|
| Elliot    | 3.8    | TensorFlow 2.13  | -        | `git+https://github.com/sisinflab/elliot.git`  |
| RecBole   | 3.9    | PyTorch >= 2.0   | <= 2.6.3 | `pip install recbole==1.2.0`                   |
| WarpRec   | 3.12   | PyTorch 2.7.0    | >= 2.54  | `pip install warprec==1.3.0`                   |

The Makefile creates all three in a single step:

```bash
make install MODE=gpu      # or MODE=cpu
```

Individual envs can also be installed on their own:

```bash
make install-elliot MODE=gpu
make install-recbole MODE=gpu
make install-warprec MODE=gpu
```

`MODE=gpu` selects the `_gpu.yml` variant of each YAML (CUDA-capable wheels); `MODE=cpu` selects the CPU-only variant.

## Running the Pipeline

The full pipeline - environment creation, data download, per-framework training, cross-framework evaluation, and table generation - is a single command:

```bash
make all MODE=gpu          # ~10 min on an A100
make all MODE=cpu          # ~45 min on a modern laptop
```

`make all` chains six idempotent stages. Each stage can also be invoked in isolation; re-running after a partial failure skips completed work.

### Pipeline stages

1. **`download`** - fetches `ml-1m.zip` from GroupLens, verifies its MD5, and converts `ratings.dat` into `data/raw/movielens.csv`.
2. **`train-warprec`** - runs WarpRec training. WarpRec owns the canonical temporal-holdout split (ratio `0.1`); it is exported to `data/split/{train,test}.tsv` so the other two frameworks evaluate against the exact same held-out interactions.
3. **`train-elliot`** and **`train-recbole`** - train ItemKNN and LightGCN in their respective frameworks against the canonical split. Recommendations are normalised to `recs/{framework}/{ItemKNN,LightGCN}.tsv`.
4. **`cross-eval`** - six cross-framework evaluations: each framework scores the other two frameworks' recommendation files via its own `ProxyRecommender` (Elliot, WarpRec) or external-evaluation script (RecBole).
5. **`aggregate`** - reads the nine canonical `metrics/train_*_eval_*.csv` files, writes `table.md`, and overwrites `table.tex` at 4-decimal precision with the same layout as the paper's version.

### Hyperparameters

Hyperparameters are frozen to the submission:

- Seed: `2020` (where applicable)
- Top-K: `10`
- ItemKNN: `k=10`, cosine similarity
- LightGCN: 2 layers, 16 factors, batch size 2048, lr 0.001, 1 epoch

The one-epoch LightGCN is intentional - the paper measures cross-framework metric agreement, not model quality.

### Inspecting targets

```bash
make help                  # list targets, pipeline flow, and current flags
```

## Outputs

| File                                 | Contents                                                                 |
|--------------------------------------|---------------------------------------------------------------------------|
| `table.md`                           | GitHub-readable dump of the reproduced table                             |
| `table.tex`                          | LaTeX table, overwritten in place                                        |
| `metrics/train_*_eval_*.csv`         | 9 canonical per-cell metric CSVs feeding the aggregator                  |
| `recs/{framework}/{ItemKNN,LightGCN}.tsv` | Normalised recommendation files used for cross-evaluation            |

## Metric Agreement Expectations

Once the top-K list is fixed, **nDCG**, **Precision**, **Recall**, **MRR**, and **MAP** are framework-independent and should match the paper's values to the last reported decimal on GPU. Minor numerical drift on LightGCN is expected on CPU.

**Gini** and **ShannonEntropy** are defined differently by each framework (normalisation and aggregation axis vary), so those columns diverge across evaluation frameworks by design. The paper body calls this out explicitly; agreement on the first five metrics with disagreement on the last two is the expected signal.

## Troubleshooting

- Ray 2.50 on macOS sometimes loses port 6379 on retry; rerunning `make train-warprec` once resolves it.
- `CUDA out of memory` on an 8 GB card during LightGCN: lower `batch_size` in `frameworks/warprec/train.yml` (and `frameworks/elliot/train.yml`) to `1024`.
- Partial state can always be recovered by rerunning the failed target; the Makefile dependency graph preserves completed artefacts.

## Cleanup

```bash
make clean                 # remove generated artefacts (data/, recs/, metrics/, ...)
make clean-envs            # destroy the three micromamba environments
```

`make clean` leaves the regenerated `table.tex` in place.
